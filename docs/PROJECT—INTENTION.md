# iching-simulator 设计与动机说明

## 项目动机

在求职 AI 全栈/全站开发工程师岗位过程中，我发现大多数 AI/Demo 项目仅停留于“接入大模型、做个壳子”，缺乏工程产品力和可延展的技术含量。**iching-simulator** 希望突破这一局限，将具备仪式感和图形丰富度的“易经起卦”仪式与现代 AI 解读、引用古籍、结构化数据流闭环有机结合，打造“可交互、可追溯、可度量、可扩展”的现代东方占卜演示平台。

本项目不仅作为兴趣驱动与工程能力提升的练习，也为面试和简历补齐“端到端 AI 产品搭建能力”、架构思维与自驱型交付的示范。

---

## 用户场景与目标用户

- 对易经/传统文化有兴趣，愿意体验数字化起卦与卦象解读仪式的普通用户
- 追求“能看得懂、可追溯依据”AI 解读的占卜体验者
- 需要可离线、保护隐私、低成本体验的技术爱好者
- 希望在技术层面参考完整 RAG/流式/前后端“全栈+多端部署”的工程师面试官

---

## 产品目标和设计原则

1. **传统仪式现代重现，强调互动感和拟真体验**
    - 前端采用 WebGL/Three.js/物理动画等手段，拟真硬币起卦/烧龟甲等仪式流程
    - 用户行为影响最终卦象，提升仪式感

2. **结构化结果与确定性起卦逻辑**
    - 保证核心起卦逻辑为纯规则算法，每次输入同参数必得一致结构化 JSON 结果
    - 便于后续分析、复盘与测试

3. **AI 解读须强制引用古籍内容，提升可信度/可追溯性**
    - 以古籍条目为 RAG 检索数据，AI 仅在引用权威文献基础上生成解释
    - 支持多流派解读风格（传统、现代行动建议等）

4. **多端部署与成本可控**
    - 支持离线前端+用户自带 key（前端直连或本地代理，保护隐私+节省成本）
    - 支持未来在线后端（配额/登录/计费），真正业务级产品演化
    - 开放 API，方便后续 CLI/Python/Node 工具或社区扩展

5. **数据闭环与可度量**
    - 支持用户“复盘打标”——记录每次解卦+后续应验情况，便于产品持续优化
    - 记录交互数据、token 消耗与引用条目，便于成本和效果分析

## 关键工程选型理由

### 前端：React+Vite+Three.js
- 社区成熟，上手快，便于 GitHub Pages 部署、样式与动画扩展
- 方便持续集成/开源及跨端技术传播

### 本地代理：Node/Python
- Node 便于前端用户快速启动，Python 便于与 AI/FastAPI 后端无缝集成
- 代理模式保障 API key 安全，规避 CORS 限制，便于日志和 token 控制

### 后续扩展：FastAPI/pgvector/Supabase
- 容易补充在线多用户、配额控制、数据可追溯、安全审计
- 向量数据库便于后续做更丰富 RAG 与检索增强
